1The maintenance gap = observability

The paper's thesis is Agent = Model + Harness, and "most agent failures are configuration failures, not model failures" (p.31). That claim only holds if you can see what the harness did. Observability is how maintenance stops being guesswork.

The components of an observability layer

The paper lists only "logs, traces, evaluations, cost and latency metering" (p.28). Here's the full set, with the two rows teams usually miss highlighted.

ComponentWhat it capturesWhy it matters for maintenance
Tracing (agent loop)One span per perceive → plan → act → observe step; nested tool calls and sub-agent hand-offs.Reconstruct what the agent did, step by step.
Trajectory capture ⭐The full sequence of tool calls + intermediate reasoning, not just the final output.Enables trajectory evals (p.22) — catching a right answer reached the wrong way.
Context / prompt snapshots ⭐What actually entered the context window each turn: rules, retrieved RAG docs, which skills fired.Most failures = "a vague rule, an absent guardrail, or a context window stuffed with noise" (p.31). You can only find these if you logged the assembled context.

⭐ = the two components missing from most setups (and from the paper's one-liner). They're what turn observability from "app monitoring" into "agent monitoring."

2Testing: deterministic tests and evals

You're doing TDD, which is great — but TDD only covers half of what the paper means by "testing." The paper's key line (p.14–15): "Without both, the practice is always vibe coding, regardless of how sophisticated the prompts are."

The two-axis model

deterministic Tests

Verify the parts that have one correct answer.

  • "Given this input, the function returns exactly that output."
  • Checked by code — pass/fail, no judgment.
  • Unit, integration, e2e, property-based.
  • This is what your TDD already does.

non-deterministic Evaluations

Verify the parts with no single right answer.

  • "Did the agent take a sane path, choose the right tools, and produce output that meets the quality bar?"
  • Checked by labelled datasets, scoring rubrics, and LM judges.
  • Output evals (the artifact) + trajectory evals (the path).
  • This is the half you're missing.
Fully deterministic — use testsFully judgment-based — use evals

Most real features live in the middle. A rule of thumb: if you can write the assertion as ==, it's a test. If you'd need a human to say "yeah, that's good enough," it's an eval.

DimensionTest (deterministic)Eval (non-deterministic)
Question it answersIs the output correct?Is the output good, and was the path sound?

ResultBinary pass/failA score / graded distribution
Good forLogic, parsing, calculations, API contracts, regressionsSummaries, classifications, RAG answers, agent tool-use, "does it look right"

Where your TDD fits (don't drop it)

TDD stays exactly as it is for deterministic logic — it's still the right discipline and the superpowers TDD skill still applies. What changes: for any feature where the output is generated, fuzzy, or AI-produced, a passing unit test is necessary but not sufficient. The paper's warning (p.34): AI errors now "look right and may even pass basic tests." Evals are what catch the failures that slip past green tests.

Mental model Tests guard the code you can pin down. Evals guard the behavior you can only describe. TDD is your test discipline; you now need an eval discipline next to it — same idea (write the check first, let it drive the work), different checker.

How to add evals — the practical loop

  1. Build a small dataset. 20–50 representative inputs with either a "golden" expected output or a rubric of what "good" means. Start tiny; grow it from real failures.
  2. Pick scorers. Mix cheap deterministic checks (regex, JSON-schema, exact-match where possible) with LM-as-judge for the fuzzy parts (relevance, faithfulness, task completion, tool correctness).
  3. Calibrate the judge. LM-as-judge alone is unreliable — spot-check its scores against your own review on borderline cases until you trust it. The 2026 consensus: calibration is what makes LM-judges scale.
  4. Gate CI on it. Run the eval suite on every change; fail the build if the score regresses past a threshold — exactly like a test.
  5. Close the loop with observability. Mine §1 production traces for new failure modes → add them to the dataset → re-run. This is the paper's "continuous quality flywheel" (p.23).

3Designing non-deterministic logic

You don't make an LLM deterministic — you contain it. Good design wraps a small non-deterministic core in deterministic shells, and validates everything that crosses the boundary. BDD and DDD each build a different wall of that box. Neither replaces the other.

The shell model

Deterministic input contractTyped inputs, ubiquitous language, schema validation
→
Non-deterministic LLM coreGenerates fuzzy output — this is what evals score
→
Deterministic output validationDomain invariants + guardrails reject bad output

DDD builds the two outer shells · evals score the core · BDD describes what the whole box should do

The one idea to keep Push the deterministic boundary inward until the non-deterministic part is as small as possible, then validate everything crossing it. A negative-total Order can never exist no matter what the model emits — because the domain boundary rejects it. Probability handles the middle; types and rules handle the edges.

Will BDD and DDD improve non-deterministic logic?

Yes — but on different walls, and neither makes the model itself more reliable. They make the system reliable around it.

BDD — the specification wall

Behavior-Driven Development

BDD's real gift isn't Gherkin syntax — it's specification by example: pin behavior down with concrete input→expected-behavior examples agreed before building.

  • A BDD scenario is an eval case. The Then clauses are your rubric.
  • The twist: swap the deterministic assertion (Then total == $42) for a graded LM-judge assertion (Then politeness ≥ 0.8).
  • Improves how you capture intent for fuzzy behavior — the weak point in evals.

DDD — the containment wall

Domain-Driven Design

DDD is the higher-leverage of the two here, because it shrinks the non-deterministic surface.

  • Domain model as a semantic filter: parse fuzzy text into typed objects; invalid domain entities can't exist — the boundary rejects hallucinations.
  • Bounded contexts decide where non-determinism is allowed (fuzzy copy = fine; payment math = deterministic code the LLM only triggers).
  • Ubiquitous language = better context engineering — the rich, grounded context the SDLC paper says agents need.
The emerging term: DICE Domain-Integrated Context Engineering — the same domain model mediates both what you send into the LLM and what you accept back. It's DDD applied to both walls at once.

Is an FAQ an eval?

An FAQ is a list of question→reference-answer pairs — which is exactly a labelled dataset, and in BDD terms, "specification by example." But a dataset isn't an eval on its own.

Eval = dataset + scorer + a run The FAQ is the dataset. Run your agent against the FAQ questions, then score its answers against the FAQ's reference answers (semantic similarity or an LM-judge). Now it's an eval. An FAQ sitting in a doc is just a fixture.

And an FAQ is one of the best starter datasets you can have — the answers are already curated, real, and human-approved. The pipeline is natural: FAQ entries → BDD scenarios → eval cases. If you have an FAQ, you're already halfway to your first eval suite.

Is DeepEval part of BDD? Marrying them in code

No — DeepEval is built on pytest (the TDD/xUnit lineage), not Cucumber/Gherkin (the BDD lineage). It's "pytest for LLM output," structurally closer to your existing TDD. But because both ride on pytest, you can compose them: write BDD scenarios with pytest-bdd, and put a DeepEval LM-judge in the Then step. You get readable, intent-first specs with non-deterministic assertions underneath.

# refund_reply.feature  — the BDD spec (readable by anyone)
Feature: Refund email replies
  Scenario: Customer requests a refund on a late order
    Given a customer email asking for a refund
    When the agent drafts a reply
    Then the reply should be polite
    And the reply should not promise a delivery date
# test_refund_reply.py  — the eval engine behind the Then clauses
from pytest_bdd import scenarios, given, when, then
from deepeval.metrics import GEval
from deepeval.test_case import LLMTestCase, LLMTestCaseParams

scenarios("refund_reply.feature")

@given("a customer email asking for a refund", target_fixture="email")
def email():
    return "My order is 2 weeks late. I want a refund."

@when("the agent drafts a reply", target_fixture="reply")
def reply(email):
    return my_agent.draft_reply(email)   # your system under test

@then("the reply should be polite")
def polite(reply):
    metric = GEval(                                  # LM-as-judge, not ==
        name="Politeness",
        criteria="The reply is courteous and non-dismissive.",
        evaluation_params=[LLMTestCaseParams.ACTUAL_OUTPUT],
        threshold=0.8,
    )
    metric.measure(LLMTestCase(input="", actual_output=reply))
    assert metric.is_successful()

BDD gives the Given/When/Then structure and readability · DeepEval's GEval scores the fuzzy clause · your TDD still guards the deterministic core inside draft_reply.

The deterministic shell in code (DDD)

The eval scores the core. The DDD shell is what makes a bad core output impossible to act on. The LLM emits fuzzy text; you refuse to let it become a domain object unless it satisfies every invariant. A hallucinated order with a negative total doesn't get "caught later" — it cannot be constructed.

# order_domain.py  — the deterministic output-validation wall
from pydantic import BaseModel, field_validator, model_validator
from decimal import Decimal

ALLOWED_CURRENCIES = {"USD", "EUR", "GBP"}

class LineItem(BaseModel):
    sku: str
    quantity: int
    unit_price: Decimal

    @field_validator("quantity")
    def qty_positive(cls, v):
        if v <= 0:
            raise ValueError("quantity must be > 0")   # invariant
        return v

class Order(BaseModel):              # the aggregate — the boundary
    currency: str
    line_items: list[LineItem]
    total: Decimal

    @field_validator("currency")
    def known_currency(cls, v):
        if v not in ALLOWED_CURRENCIES:
            raise ValueError(f"unsupported currency: {v}")
        return v

    @model_validator(mode="after")
    def total_matches_lines(self):
        computed = sum(li.unit_price * li.quantity for li in self.line_items)
        if not self.line_items:
            raise ValueError("order must have at least one line item")
        if computed != self.total:                    # cross-field invariant
            raise ValueError(f"total {self.total} != computed {computed}")
        return self
# the containment loop — fuzzy in, validated-or-rejected out
from pydantic import ValidationError

def order_from_agent(email: str, max_retries: int = 2) -> Order:
    for attempt in range(max_retries + 1):
        raw = llm.extract_order(email)            # non-deterministic core
        try:
            return Order.model_validate_json(raw)    # shell accepts ONLY valid
        except ValidationError as e:
            # feed the exact error back — the SDLC paper's self-correct loop
            email = f"{email}\n\nYour last output was invalid: {e}"
    raise ValueError("agent could not produce a valid Order")  # fail closed

No matter what the model emits, downstream code only ever receives a fully-valid Order. The non-deterministic surface shrinks to "did extraction succeed within N tries?" — and the ValidationError becomes the feedback signal that drives the agent's retry loop.

Why this is the higher-leverage move The eval tells you how often the core is good (a quality metric). The DDD shell guarantees that bad output never reaches production (a safety guarantee). You want both — but if you can only build one first, build the shell. A failing eval is a dashboard problem; an unvalidated hallucinated order is an incident.

4A starter path that fits how you work

You drive everything by prompt + Claude Code. So pick tools with MCP servers and self-hosting, and let the assistant wire them. Smallest first step that closes the most gap:

  1. Stand up Langfuse (self-hosted) + OpenTelemetry. Instrument one project. Now every agent run has traces, token cost, and latency. Add the Langfuse MCP server so you can ask "show me the slowest run today" by prompt.
  2. Add Sentry for app errors + MCP-server monitoring — one line, immediate prod visibility, debuggable from the editor.
  3. Add DeepEval next to your existing tests. Write 20 eval cases for your fuzziest feature. Gate CI on the score. You now have both halves of the paper's testing definition.
  4. Wire the flywheel: a weekly prompt — "pull last week's Langfuse traces, cluster failures, propose new DeepEval cases." Observability feeds evals; evals catch drift; drift shows up in observability. That loop is the maintenance phase the paper says you're missing.
One brainstorming session away When you're ready, run your usual brainstorming skill scoped to "add observability + evals to <project>" and hand it this doc. It'll turn step 1–4 into a concrete plan for your actual codebase.
